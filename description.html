Selection Attributes Catalogue

Description

Here we present the selection attributes identified in the catalogue with their description.

Adequacy criterion
Test case generation and stopping rule.
Algorithm
Incorrect or missing implementation that can be fixed by reimplementing an algorithm or data structures.
Application Category
Information on what type of application the approach can be used.
Aspect
Functionality of the system to be tested.
Assignment
Values assigned incorrectly or not assigned.
Automation
Part of the technique automated by the tool.
Behavioural/Structural Model provided
Indicates the models proposed in scientific research.
Benefits
Benefits of using the technique.
Build/package/merge
Describe mistakes in library systems, management of changes, or version control.
Checking
Missing or incorrect validation of data or incorrect loop or conditional statements.
Completeness
Coverage provided by the set of cases.
Comprehensibility
Whether or not the technique is easy to understand.
Context/Objective of the concurrent software
Description of the context/objective of the software.
Cost
Cost of tool purchase and maintenance.
Cost of application
How much effort it takes to apply the technique.
Dependencies
Relationships of one technique with another.
Development method
Development method or life cycle to which it is linked.
Development paradigm
Indicates the delopment paradigm used in the software project.
Documentation
This defect type affects both publication and maintenance notes, it has a significant meaning only in the early stages of software life cycle (specification and high level design).
Effectiveness
What capability the set of cases should have to detect defects.
Effort required
Indication of the effort, time and cost to execute the tasks in using the technique.
Element
Elements of the system on which the test acts.
Environment
Platform (software and hardware) and programming language with which the tool operates.
Experience
Experience required to be able to apply the technique.
Experimental Evaluation
Indication of the experimental strategy and evidence obtained in the evaluation.
Function
Affects a sizable amount of code and refers to capability that is either implemented incorrectly or not.
Identifier
Name of the tool and the manufacturer.
Inputs
Inputs required to apply the technique.
Interface
Errors in the interaction among components, modules, call statements, parameters list.
Knowledge
Knowledge required to be able to apply the technique.
Mechanism that execute different synchronization sequences
Approach that forces the execution of different synchronization sequences.
Modelling Technology
Indication of the technology used for modelling.
Number of generated cases
Number of cases generated per software size unit.
Opinion
General opinion about the technique after having used it.
Outputs
Information about the outputs (artifacts) produced by the approach.
Personnel
Personnel who worked on earlier projects.
Precision
How many repeated test cases the technique generates.
Problems
Problems with using the technique.
Program representation
Strategie to capture relevant information for testing.
Programming language
Programming language with which it can be used.
Purpose
Type of evaluation and quality attribute to be tested in the system.
Reference projects
Earlier projects in which the technique has been used.
Repeatability
Whether two people generate the same test cases.
Reusability of test cases
Indication about the reusability of test cases.
Size
Size that the software should have to be able to use the technique.
Software architecture
Development paradigm to which it is linked.
Software Quality Characteristics
Indication of the set of quality characteristics that the approach is capable of evaluating.
Software type
Type of software that can be tested using the technique.
Sources of information
Where to find information about the technique.
Support
Support provided by the tool manufacturer.
Technique that treated the state explosion problem
Indicates the technique that treated the problem of state explosion.
Test data cost
Cost of identifying the test data.
Testing Level
The test levels, explored by the approach to test software behavior/structure.
Time when the basic functionality is exposed
Indication about the time when the basic functionality is exposed.
Timing/serialization
Missing or incorrect serialization of shared resources.
Tools used
Tools used in earlier projects.
Type of Analysis
Indication about the type of test analysis that the approach is able to support: static behavior (not influenced by user interaction) or dynamic (eg, pages that can be dynamically constructed in response to user input).
Type of faults identified
Type of faults that the approach can help reveal.
Type of replay mechanism
Indication about the type of replay mechanism used for the re-execution of the program during testing.
Type of Testing Technique
Indication of the test technique that the approach supports.
Unity
Describes the unit of the application in which the approach works, such as Web pages, Web page components, links, and others.
